On-Premise

서버를 클라우드 같은 원격 환경에서 운영하는 것이 아닌 자체적으로 보유한 전산실 서버에 직접 설치해 운영하는 방식

클라우드 컴퓨팅 기술이 나오기 전까지 일반적인 방법임.

장점

  • 비지니스 정보를 보안성 높게 관리 가능
  • 반대로 직접 가지고 있는 것은 위험할 수 도 있음(현금을 집에 보관하기보다 은행에 맡기는게 나은 것처럼..)

단점

  • 시스템 구축에 시간이 많이 걸리고, 비용이 많이 듦

참고)

  • Off-premise가 클라우드 방식의 서비스 방식을 뜻함
  • premise는 전제, 건물, 토지, 선행조건을 뜻함

PaaS

Platform as a Service의 약자로, 클라우드 컴퓨팅 서비스.

애플리케이션을 구축, 실행 및 관리하는 데 필요한 플랫폼을 제공하는데 개발자가 애플리케이션 코드를 작성하고 실행 환경, 데이터베이스, 웹 서버 등의 기술적인 측면을 고려하지 않고도 애플리케이션을 배포할 수 있게 도와준다.

IaaS(infrastructure as a Service)보다 더 추상화된 레벨에서 작동하며, 사용자는 서버, 스토리지, 네트워크 등의 하드웨어 자원에 대해 걱정할 필요가 없다.

ex) AWS Elastifc Beanstalk, heroku

장점

  • 인프라 및 관리 비용을 절감 : 하드웨어 인프라를 구매할 필요가 없고, 애플리케이션 운영 및 관리를 자동화 할 수 있음
  • 개발 생산성 높일 수 있음 : 애플리케이션 코드 작성에만 집중
  • 애플리케이션을 쉽게 확장

단점

  • 제한된 제어 수준
  • 보안문제
  • 공급업체에 대한 의존도
  • 제한된 커스터마이징
  • 비용 예측의 어려움

참고)

  • https://www.samsungsds.com/kr/cloud-glossary/paas.html

IaaS

Infrastructure as a Service

종량제 서비스로, 제 3사가 스토리지와 가상화 같은 인프라 서비스를 인터넷을 통해 클라우드로 제공 사용자는 운영 체제 및 데이터, 애플리케이션, 미들웨어 및 런타이을 담당 제공업체는 사용자가 필요로 하는 네트워크, 서버, 가상화 및 스토리지의 관리와 액세스를 담당

장점

  • 필요한 구성 요소만 구매하고 필요에 따라 확장 또는 축소할 수 있는 유연성 제공
  • 개발 및 테스트 환경의 구축 및 제거가 빠르고 유연

단점

  • 제공업체의 보안 문제 가능성

ex) AWS, Microsoft Azure, Google Cloud 같은 퍼블릭 클라우드 공급 업체가 IaaS의 예시

참고)

  • https://www.redhat.com/ko/topics/cloud-computing/iaas-vs-paas-vs-saas

SaaS

Software as a Service

클라우드 애플리케이션과 기본 IT인프라 및 플랫폼을 인터넷 브라우저를 통해 최종 사용자에게 제공하는 클라우드 컴퓨팅 형태

장점

  • 소프트웨어를 영구 구매하거나 강력한 온프레미스 IT 인프라에 투자할 필요가 없어서 초기 비용이 줄어듦

단점

  • 서비스 성능은 인터넷 연결 속도에 따라 달라지므로 고객은 고속 네트워크 하드웨어에 투자해야 함

ex) Google Docs, Microsoft Office 365 같은 서비스 저공업체와 인사 관리 소프트웨어, 통합개발환경(IDE)를 제공하는 엔터프라이즈 서비스

참고)

  • https://www.redhat.com/ko/topics/cloud-computing/what-is-saas
  • 사진으로 보는 비교

favicon

파비콘 또는 패비콘이란 인터넷 웹 브라우저의 주소창에 표시되는 웹사이트나 웹페이지를 대표하는 아이콘

사용 방법

<!-- Chrome, Safari, IE -->
<link rel="shortcut icon" href="path/to/favicon.ico">
 
<!-- Firefox, Opera (Chrome and Safari say thanks but no thanks) -->
<link rel="icon" href="path/to/favicon.png">

ico 파비콘은 Multi sizes를 지원하기에 하나의 ICO 파일에 여러 사이즈의 아이콘을 넣어서 사용 가능 하지만, PNG는 불가능하기 때문에 사이즈를 모두 선언해야한다.

ex)

<link rel="icon" href="favicon-16.png" sizes="16x16">
<link rel="icon" href="favicon-32.png" sizes="32x32">
<link rel="icon" href="favicon-48.png" sizes="48x48">
<link rel="icon" href="favicon-64.png" sizes="64x64">
<link rel="icon" href="favicon-128.png" sizes="128x128">

PNG 사용 시 파비콘 결정 방법

  • FireFox, Safari는 마지막에 제공되는 파비콘 사용
  • 맥용 Chrome은 ICO가 아니라면 32x32 사용
  • 윈도우용 Chrome은 16x16이 먼저 선언되지 않는다면 ICO 사용

Open Graph Protocol

2010년에 페이스북에서 처음 소개

해당 프로토콜을 사용하면 웹페이지가 소셜 그래프에서 보기 좋게 표현될 수 있다.

HTML 문서의 메타정보를 쉽게 표시하기 위해서 메타 정보에 해당하는 제목, 설명, 문서의 타입, 대표 URL 등 다양한 요소들에 대해서 사람들이 통일해서 쓸 수 있도록 정의해놓은 프로토콜

og meta 태그에는 url, image path에 상대경로가 불가능하다.

기본적으로 해당 프로토콜을 지켜서 보여지는데 문제는 없겠지만, 각 서비스마다 권장하는 값들이 있음

Facebook

  • title - 40자 권장(모바일)
  • image
    • 최소 크기 200x200
    • 1200 x 630 권장
    • 1.91 : 1 비율 권장
    • 8MB 제한
    • JPG, GIF, PNG

Twitter

  • title - 55자 권장, 70자 제한
  • description - 200자 제한
  • image
    • 최소 크기 200x200
    • 1200 x 630 권장
    • 1.91 : 1 비율 권장
    • 5MB 미만
    • JPG, PNG

ex) 카카오톡이나 디스코드, 페이스북에서 웹 사이트 링크를 공유하면 링크가 아니라 게시물처럼 보이게 되는 것

Chrome 리소스 로드 우선 순위

  • CSS와 폰트들은 가장 높은 우선 순위를 가짐
  • 스크립트는 문서에서 어느 위치에 있는지에 따라 다름
    • async, defer, blocking(일반적인 스크립트 포함 방법) 적용여부에 따라 다른 우선순위\
  • 뷰포트에 표시되는 이미지는 뷰포트에 보이징 않는 이미지보다 우선순위 높음

참고)

  • async
    • 비동기 스크립트
    • 페이지와 완전히 독립적으로 동작
    • defer와 마찬가지로 백그라운드에서 다운
    • HTML 페이지는 async 스크립트 다운이 완료되길 기다리지 않고 페이지 내 콘텐츠를 처리, 출력
      • 하지만 async 스크립트 실행중에는 파싱이 멈춤
    • DOMContentLoaded 이벤트와 async 스크립트는 서로 기다리지 않음
    • async는 defer와 달리 다운로드가 끝난 스크립트 순으로 실행
    • 추가) dynamic imports는 기본적으로 async처럼 동작
  • defer
    • 지연스크립트
    • 백그라운드에서 다운
    • HTML 파싱이 멈추지 않음
    • DOM이 준비된 후에 실행되긴 하지만 DOMContentLoaded 이벤트 발생 전에 실행
    • 지연 스크립트도 일반 스크립트와 마찬가지로 HTML 추가된 순으로 실행
